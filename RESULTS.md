# Пост-фикс keepalive 5→75 + firewall — регрессия? (2026-06-23)

**Вопрос:** не просели ли throughput/latency/error-rate после `keepalive_timeout 5→75`
(фикс интермиттентной CF-edge 500 на signup) и ufw-файрвола origin? И атрибутируются
ли 5xx краем, а не origin?

**Метод:** полный свип через **домен Cloudflare** (`--target prod`, как и старые
таблицы — apples-to-apples), duration 10s, warmup 2s, `--cleanup`. Бенч дополнен
атрибуцией 5xx по заголовкам `X-App`/`X-Origin` (см. CLAUDE.md). Маркеры подтверждены
живьём: genuine Django-ответ несёт оба маркера.

**Ответ: регрессии нет — стало заметно ЛУЧШЕ.** За ~18 600 запросов — **0 HTTP 5xx**
любого класса (app/infra/edge). Единственная «ошибка» (c=40 sync, 0.1%) — клиентский
таймаут-эксепшен, не серверная 500 (строки `5xx attrib`/`[ERROR]` не появились).

## Выводы по фиксу

1. **Никакой регрессии от keepalive 75 / firewall.** Latency сопоставима, error-rate
   ≤ прежнего (в основном 0%), throughput не просел — наоборот, Django sync впервые
   вышел на ~160 RPS (исторически плато было ~65; sync 160 для Django — впервые,
   тогда как heartbeat ~160 был и раньше). **Причина роста не доказана** (контрольного
   A/B keepalive 5↔75 не делали). Атрибуция «keepalive» сомнительна: `keepalive_timeout` —
   это клиент-facing keepalive CF↔nginx, общий для всех бэкендов; Go ходил через тот же
   CF/nginx и выдавал ~490, т.е. фронт-прокси НЕ резал throughput до 65 — потолок
   Django в 65 был, скорее, его собственный (worker-bound). К тому же keepalive-5 бьёт
   по *idle-reuse* (редкая CF-edge 500), а не по непрерывному closed-loop RPS. Что
   именно подняло Django до 160 — открытый вопрос; для целей этой задачи (поиск
   регрессии) не важно.
2. **Опасение про исчерпание FD/соединений от длинного keepalive — не подтвердилось.**
   Если бы idle-коннекты копились, на высоком c полезли бы infra-5xx / connection-
   ошибки. Вместо этого c=80/100/160 — чисто, 0 ошибок.
3. **Атрибуция 5xx работает и показала 0 origin-500.** За весь свип ни одного 5xx
   (app/infra/edge). Бенч теперь печатает `5xx attrib: app=N infra=N edge=N` при
   любой 500 — будущая регрессия классифицируется автоматически.

# Результаты нагрузочного тестирования — prod (2026-06-14)

Замеры ёмкости прод-бэкенда `tasks.titov.dev`, чтобы не пере-обсуждать «слабый ли
сервер». Кратко: **нет, не слабый для текущей стадии — медиана отличная, потолок
адекватный, единственная слабость — редкий хвостовой джиттер дешёвого хоста.**

## Стенд
- **Сервер:** RackNerd VPS, **$20/год**. ~4 vCPU (shared/oversold), 2 ГБ RAM,
  Django + gunicorn (~8 sync-воркеров) + Postgres 18.
- **Клиент:** Windows (Xeon E5-2678 v3, 24 logical), `bench_rest.py` (closed-loop,
  нулевая пауза между запросами).
- **Метод:** свип concurrency, duration 10s, warmup 2s, `--cleanup` на каждый
  прогон. На Windows обязательно `PYTHONUTF8=1` (см. ниже).

> ⚠️ **concurrency ≠ юзеры.** В бенче c=100 — это 100 запросов В ПОЛЁТЕ
> ОДНОВРЕМЕННО, безостановочно. Реальный юзер шлёт запрос раз в секунды/минуты.
> c=100 closed-loop ≈ **тысячи** реальных активных юзеров.

> ⚠️ **Latency здесь = в основном сетевой RTT, а НЕ время сервера.** Сервер в
> США, клиент — на другом конце планеты, так что базовые ~185мс p50 — это
> intercontinental round-trip, а серверное время вычисления — десятки мс.
> Все цифры latency ниже несут этот RTT-налог; у юзеров рядом с сервером (США)
> будет кратно меньше. Чтобы померить «чистое» серверное время — гонять бенч
> из того же региона (US-бокс / сам сервер).

## heartbeat (`POST /api/auth/heartbeat` — оверхед фреймворка, базовая latency)

|----|------|-------|-------|--------|--------|--------|
| 10 | 50.6 | 180мс | 197мс | 482мс | 1863мс | 0% |
| 50 | 85.0 | 159мс | 190мс | 724мс | 18471мс | 0.3% |
| 100 | **152** | 197мс | 327мс | 705мс | 8419мс | 0.2% |
| 200 | **161** | 179мс | 538мс | 10250мс | 15751мс | 0.1% |

→ потолок «лёгкого» эндпоинта **~150-160 RPS**, медиана здоровая даже при c=200.

|-----|------|-------|--------|---------|---------|--------|
| 1 | 5.4 | 183мс | 248мс | 265мс | 265мс | 0% |
| 5 | 15.6 | 184мс | 222мс | 4272мс | 7571мс | 0% |
| 10 | 22.0 | 187мс | 216мс | 498мс | 14008мс | 0% |

→ **throughput-потолок sync ≈ 65 RPS** (плато с c=80). **Колено c=40-80**:

## Чистый хвост при c=1 (без конкуренции) — что джиттерит

Прогон по 120с, c=1 (один тёплый keep-alive коннект), htop сервера при этом спит
(load 0.01):

| сценарий | reqs | p50 | p95 | p99 | max |
|----------|------|-------|-------|-------|--------|
| heartbeat (без БД) | 644 | 184мс | 190мс | 236мс | **741мс** |
| sync (6 таблиц) | 551 | 205мс | 224мс | 536мс | **2360мс** |

→ **На c=1 хвост чистый** (max 0.7-2.4с) — никакого idle-джиттера на секунды.
→ **БД дешёвая:** sync добавляет к медиане всего **~20мс** над heartbeat за
6-табличный запрос; хвост чуть толще, но секунды максимум.
→ **Многосекундные выбросы (4-14с) видны только при c≥5, отсутствуют при c=1**
— то есть они **коррелируют с concurrency, а не с простоем**. c=5 < 8 воркеров,
очереди быть не должно → вероятная причина — **установка новых TLS-соединений
через океан** (на c=1 один коннект переиспользуется, хвоста нет; при c≥5 новые
коннекты, каждый — handshake+RTT через полпланеты). Это **сеть/путь, а не
сервер/Postgres.**

## Выводы

1. **Сервер адекватен.** ~50-65 RPS sync / ~150 RPS heartbeat. p50 ~185мс — это
   **в основном RTT до США**, серверное время кратно меньше. При ~1 синке на
   юзера в 30-60с это **сотни-тысячи одновременно активных**. Для пет-проекта/
   раннего запуска — с огромным запасом. За $20/год — отличная сделка.
2. **Узкое место под burst — число воркеров (~8), не CPU/БД и не пропускная.**
   За порогом concurrency запросы встают в очередь, RPS не растёт, растут хвосты.
3. **Многосекундные хвосты коррелируют с concurrency, НЕ с простоем** (см. секцию
   c=1: при одном тёплом коннекте max 0.7-2.4с, выбросов 4-14с нет; они появляются
   с c≥5). c=5 < 8 воркеров → это не очередь воркеров и не Postgres (sync при c=1
   лишь +20мс к медиане). Вероятная причина — **установка новых TLS-коннектов
   через океан** под concurrency (handshake+RTT на каждый новый коннект). То есть
   **сеть/длинный путь, а не сервер/БД.** (Старая версия «idle-джиттер хоста»
   опровергнута замером c=1.)
4. **Про флаку signup в интеграционных тестах:** это **не нагрузка** (4
   одновременные регистрации = нагрузка уровня c≈4, сервер при c=50 отдаёт
   p50=159мс — никак не 10с) и **не медленный сервер** (c=1 чистый). Правдоподобно
   — редкий **хвост установки соединения через океан** под лёгкой concurrency
   (п.3), в который signup попадал по невезению. 20-секундный полл ожидания
   `Set Passcode` это гасит. (Доказательно не докопано — для пруфа нужна репро с
   коротким поллом + лог времени ответа `POST /api/auth/signup` в упавшем прогоне.)
5. **Когда апгрейдить хост:** когда устойчивый трафик подойдёт к ~50 RPS sync,
   или хвостовой джиттер начнёт бить по UX живых юзеров, или 2 ГБ RAM зажмёт
   (больше Postgres-коннектов). Это «проблема успеха» — не сейчас.

## Фикс бенча (нужен, иначе не запустится на Windows)

Коммит `1a71162` в этом репо:
- **Per-request timeout guard** (`aiohttp.ClientTimeout(total=20)`): без него дефолт
  aiohttp = 5 мин, и один застрявший коннект под насыщением вешает весь
  `asyncio.gather` на ~300с (выглядит как зависший бенч). С гардом столл = ошибка,
  воркер продолжает, error-rate становится честным сигналом насыщения.
- **UTF-8 вывод**: box-drawing символы (`─`) крашат бенч на Windows-консоли
  (cp1252). `sys.stdout.reconfigure(encoding="utf-8")`.

## Как воспроизвести

```bash
cd tasks_benchmark
python -m venv venv && ./venv/Scripts/python -m pip install -r requirements.txt   # Windows
# свип sync:
for c in 1 5 10 20 40 80 160; do
  PYTHONUTF8=1 ./venv/Scripts/python -u bench_rest.py --target prod --scenario sync \
    --concurrency $c --duration 10 --cleanup
done
```
На macOS/Linux `PYTHONUTF8=1` не нужен (консоль уже UTF-8).
